# Спецификация: Интернет-магазин автозапчастей «Sacramento» — v1.0

> Дата: 2026-06-10. Статус: **согласовано** (все открытые вопросы закрыты, решения — в разделе 14).

## 1. Обзор

Интернет-магазин автомобильных запчастей (Молдова, валюта продажи — MDL, молдавский лей)
с подбором по VIN-коду, заказом с доставкой курьером или самовывозом, и админ-панелью.
Каталог ~5 000 SKU (по факту текущего файла наличия — 4 940 позиций).

**Стек:**
- Бэкенд: Java 25, Spring Boot 4, Spring Security (логин/пароль для админки),
  Spring Data JPA, Flyway
- БД: PostgreSQL 17
- Витрина: **Next.js + MUI** (SSR — для SEO), хостинг Vercel
- Админка: React + Vite + MUI (SPA), раздаётся с VPS
- Excel/CSV: Apache POI + OpenCSV
- Деплой бэкенда: DigitalOcean VPS $6 (1 vCPU / 1 GB), **без Docker** —
  PostgreSQL и nginx из apt, бэкенд под systemd; цель — **GraalVM native image**

## 2. Архитектура и деплой

```
[Покупатель] → Vercel: Next.js витрина (SSR, SEO) ─┐
[Админ]      → VPS: nginx → /admin (React SPA)     ├→ REST API: Spring Boot (VPS, systemd)
                                                    └→ PostgreSQL (VPS, apt)
```

- Монолитный бэкенд, модули по пакетам: `catalog`, `vehicles`, `pricing`,
  `orders`, `importexport`, `media`, `auth`.
- Домен: покупается; витрина — `sacramento.md` (условно), API — `api.sacramento.md`,
  админка — `admin.sacramento.md` или `/admin` на VPS.

### Укладываемся в 1 GB RAM (дроплет $6, без Docker)

Docker не используем — на VPS всё ставится нативно, это экономит ~100–150 МБ
и упрощает обслуживание одного сервера:

| Компонент | Как ставим | Бюджет RAM |
|---|---|---|
| PostgreSQL 17 | apt, тюнинг: `shared_buffers=128MB`, `max_connections=20` | ~150–200 МБ |
| Бэкенд | **GraalVM native image** (Spring Boot 4 AOT), systemd-сервис | ~150–250 МБ |
| nginx | apt (TLS Let's Encrypt, статика админки, прокси API) | ~10 МБ |
| swap 2 GB | страховка от OOM | — |

- **GraalVM native image** — основной план: Spring Boot 4 официально поддерживает
  AOT-компиляцию; нативный бинарник стартует за десятки миллисекунд и потребляет
  в 2–3 раза меньше памяти, чем JVM.
- ⚠️ Риск: Apache POI в native image требует reachability-метаданных (рефлексия).
  Проверяем экспорт/импорт в native-сборке на раннем этапе. **Фолбэк**: обычный
  JVM-jar (`-Xmx384m -XX:+UseSerialGC`) — тоже влезает в 1 GB со swap.
- Сборка native image требует ~8 ГБ RAM — собираем в CI (GitHub Actions, бесплатно)
  или локально, на VPS заливается готовый бинарник.

## 3. Роли и аутентификация

- **Гость** — каталог, VIN-поиск, корзина, оформление заказа. Личного кабинета нет,
  регистрация покупателей не нужна. Заказ оформляется по имени + телефону (+e-mail опц.).
- **Админ** — единственная роль в админке. Spring Security, логин/пароль (BCrypt),
  сессии, защита от перебора (задержка/lockout), CSRF.

## 4. Каталог

- **Категории** — дерево (parent_id), CRUD в админке.
- **Товар**: артикул (SKU), название, бренд (опц.), описание, OEM/кросс-номера
  (множественные — в исходном файле встречаются через `/`), категория,
  закупочная цена + валюта закупки (USD или MDL — как в учёте), розничная цена
  (расчётная или ручная), **оптовая цена**, остаток, место на складе (полка,
  формат «43*19» из файла), заметка для админа (внутренняя), фото, признак «активен».
- Поиск: название, артикул, OEM-номер; pg_trgm (триграммный индекс) — терпимость
  к опечаткам и частичному совпадению.
- Фильтры: категория, бренд, наличие, цена, выбранный автомобиль (после VIN).

### Фото
- Хранение **в PostgreSQL** (bytea): оригинал + сгенерированный thumbnail (WebP).
- Несколько фото на товар, одно — главное.
- Настройки в админке: максимальный размер загрузки, качество сжатия.
- Отдача через API с HTTP-кэшированием (ETag, Cache-Control) — критично для слабого VPS.

## 5. Подбор по VIN

VIN идентифицирует автомобиль, а не запчасти, поэтому схема двухступенчатая:
`VIN → автомобиль → товары в наличии для него`.

### 5.1. Декодирование VIN — бесплатная стратегия (MVP)

Полностью бесплатного декодера с хорошим покрытием европейского рынка не существует
(Laximo/TecDoc — платные). Поэтому **гибрид из трёх бесплатных уровней**:

1. **Локальный декодер (своя таблица, 0 запросов, 0 ₽):**
   - WMI (первые 3 символа) → производитель (таблица WMI публична);
   - 10-й символ → модельный год;
   - валидация контрольной цифры (9-й символ).
2. **NHTSA vPIC** (https://vpic.nhtsa.dot.gov/api/) — государственный API США:
   полностью бесплатный, без ключа и без лимитов. Для европейских VIN часто
   возвращает марку, год, иногда модель/двигатель. Используем как обогащение.
3. **Подтверждение пользователем:** после декодирования показываем
   «Ваш автомобиль: Audi, 2016. Уточните модель» — и предлагаем выбор из **нашего**
   справочника автомобилей (только те, для которых есть товары). Один клик.

Архитектурно — интерфейс `VinDecoder` с цепочкой адаптеров
(`LocalWmiDecoder → NhtsaDecoder → ManualConfirm`), чтобы при росте бизнеса
подключить Laximo/платный API без переделки.

Обязательно: `vin_cache` (VIN → результат, повторные запросы бесплатны),
rate-limit на эндпоинт (например, 10 запросов/мин с IP).

### 5.2. База применимости

В исходном файле применимость зашита свободным текстом в название
(«Audi A4 8/97>01 радиатор кондиционера»). Поэтому:

- Таблицы: `vehicle` (марка, модель, годы от/до, двигатель опц.)
  и `product_vehicle` (many-to-many).
- **Первичный импорт**: парсер эвристически извлекает марку/модель/годы из названия
  (словарь марок: Audi, VW, Toyota, Porsche, Kia…; паттерны лет: «8/97>01», «11 - 16»,
  «2012»). Что распозналось — связывается автоматически с пометкой «авто-распознано».
- **Админка**: экран «Применимость» — список нераспознанных товаров, массовая привязка
  товаров к автомобилям, редактирование справочника автомобилей.

### 5.3. Поток на витрине

1. Поле VIN на главной (17 символов, без I/O/Q, валидация контрольной цифры).
2. Декодирование → подтверждение модели (один клик) → бейдж «Ваш автомобиль: …»
   закрепляется в шапке (cookie).
3. Каталог фильтруется: товары, привязанные к этому авто, в наличии.
4. VIN не распознан → ручной выбор: марка → модель → год.

## 6. Цены, курс, опт и скидки

- Закупочная цена хранится в валюте закупки (USD или MDL).
- **Курс**: ежедневный планировщик тянет официальный курс **Национального банка
  Молдовы (BNM)** — XML API `https://www.bnm.md/ro/official_exchange_rates?get_xml=1&date=…`.
  Таблица `exchange_rate` (валюта, курс, источник BANK/MANUAL, дата) + история.
- В админке: переключатель «курс BNM / ручной», поле ручного курса.
- **Розничная цена** = закупка × курс × (1 + наценка%) → округление (правило
  настраивается: до 1 лея / до 5 леев). Наценка — глобальная в админке +
  переопределение на категорию и на товар. Возможна и полностью ручная цена.
- **Оптовая цена** — отдельное поле/наценка; на витрине не показывается,
  видна только админу; админ вручную переключает заказ на оптовые цены.
- **Скидки** — только ручная скидка на заказ в админке (% или сумма),
  промокодов на витрине нет.
- При изменении курса розничные цены пересчитываются автоматически (кроме ручных).

## 7. Корзина и заказ

- Корзина гостя — localStorage, отправляется на сервер при оформлении.
- Оформление: имя, телефон (обязательно), e-mail (опц.), способ получения,
  способ оплаты, комментарий.
- Статусы: `NEW → CONFIRMED → ASSEMBLING → (DELIVERING | READY_FOR_PICKUP) → DONE`,
  из любого — `CANCELLED` (с причиной).
- **Резервирование остатка — при подтверждении заказа админом** (NEW → CONFIRMED).
  В заказе фиксируются цены на момент оформления.
- Уведомления — **только в админку**: счётчик новых заказов в шапке админки
  (поллинг каждые 30 с), список на dashboard. Без SMS/e-mail/Telegram.

## 8. Доставка и оплата

| Получение | Оплата | Стоимость |
|---|---|---|
| Курьер (свой) | наличными курьеру | фиксированная (настройка в админке) |
| Самовывоз (одна точка) | карта на POS-терминале или наличные | бесплатно |

Онлайн-эквайринга нет — способ оплаты фиксируется в заказе, деньги принимаются офлайн.
Адрес и часы работы точки самовывоза — настройки в админке, показываются при оформлении.

## 9. Админка

- **Dashboard**: новые заказы (счётчик), заказы за сегодня, товары с нулевым остатком,
  текущий курс.
- **Товары**: таблица с поиском/фильтрами, создание/редактирование (вкладки: основное,
  цены, фото, применимость, склад), массовые операции (активировать/деактивировать).
- **Категории**: дерево, CRUD, сортировка.
- **Применимость**: справочник автомобилей, очередь нераспознанных товаров.
- **Заказы**: список, фильтр по статусу, карточка заказа, смена статуса, скидка/опт,
  печатная форма.
- **Курс и цены**: курс BNM/ручной, история, глобальная наценка, правило округления.
- **Импорт/экспорт**: см. раздел 10.
- **Настройки**: стоимость доставки, точка самовывоза, параметры фото, смена пароля.

## 10. Экспорт / импорт (снэпшот базы)

### Экспорт — ежедневный снэпшот
- `@Scheduled` cron 23:55: полный снэпшот каталога в CSV (UTF-8 BOM, `;`) и XLSX:
  артикул, название, бренд, категория, OEM-номера, закупочная цена + валюта,
  розничная цена, оптовая цена, остаток, место на складе, применимость, активность.
- Хранение последних N снэпшотов (настройка, по умолчанию 30) в БД/на диске,
  страница «Экспорты» в админке: история + скачивание + кнопка «Выгрузить сейчас».

### Импорт — загрузка снэпшота обратно
1. Загрузка CSV/XLSX → парсинг → валидация (обязательные поля, типы, дубли артикулов).
2. Предпросмотр: N создаётся / M обновляется / K с ошибками (с указанием строк).
3. Подтверждение → транзакционный upsert по артикулу.
4. Отчёт + скачиваемый файл ошибочных строк.

### Первичный импорт из текущего учётного файла (.xls)
Отдельный одноразовый сценарий: маппинг колонок текущего файла
(название → парсинг применимости; артикул; количество; закупка USD/MDL;
полка), очистка мусорных строк. Закупочные цены и суммы прихода — только
в админку, на витрину не попадают.

## 11. SEO (витрина)

- SSR на Next.js: товары и категории рендерятся на сервере.
- ЧПУ-URL: `/catalog/<категория>/<артикул>-<слаг>`.
- `sitemap.xml` (автогенерация), `robots.txt`, meta title/description по шаблону,
  Open Graph, микроразметка schema.org `Product` (цена, наличие).
- Язык — русский (один).

## 12. Безопасность

- Админка — только под логином (Spring Security, BCrypt, сессии).
- HTTPS везде (Let's Encrypt на VPS, Vercel — из коробки).
- Rate-limit: VIN-эндпоинт, оформление заказа, логин.
- Bean Validation на всех входных DTO; загрузка файлов — лимит размера, проверка типа.
- CORS: API принимает только домены витрины и админки.
- Бэкап PostgreSQL: ежедневный `pg_dump` + ежедневный CSV-снэпшот (раздел 10).

3. Первичный импорт .xls + парсер применимости.

5. Корзина, оформление, заказы в админке.
6. VIN-подбор (локальный декодер + vPIC + подтверждение).
7. Экспорт/импорт снэпшотов.
8. SEO, безопасность, native image, деплой на VPS + Vercel.

## 14. Принятые решения (бывшие открытые вопросы)

1. **Опт** — админ вручную переключает заказ на оптовые цены; оптовые цены видны
   только в админке.
2. **Скидки** — только ручная скидка на заказ в админке, промокодов нет.
3. **Валюта закупки** — хранится в валюте закупки (USD или MDL) с пометкой.
4. **Место на складе** — поле товара, видно только админу.
5. **VPS** — дроплет $6 (1 GB), без Docker, GraalVM native image + swap;
   фолбэк — JVM-jar с ужатым хипом; апгрейд до $12 при необходимости.
6. **Витрина** — Next.js + MUI (SSR) на Vercel; админка — React + Vite SPA на VPS.
